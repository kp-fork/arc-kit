# OKF Compatibility Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans or superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add Open Knowledge Format (OKF) compatibility to ArcKit without replacing the existing ARC artifact model. ArcKit remains the source of truth for architecture governance; OKF becomes a portable export/import and metadata layer.

**Architecture:** Three-phase implementation. First, add deterministic OKF metadata helpers and an export command that produces conformant OKF bundles from existing `projects/` artifacts. Second, add optional OKF frontmatter stamping for newly written ARC artifacts, with hooks and manifest generation taught to preserve and read it. Third, add an import command that ingests external OKF bundles into ArcKit research, tech-note, vendor-profile, or candidate artifact workflows.

**Tech Stack:** Claude Code plugin markdown commands/templates, Node ESM hooks and helper modules, Python/Node test suites, `scripts/converter.py` for generated extensions, markdown docs and site updates.

**Primary references:**
- Google Cloud OKF announcement: `https://cloud.google.com/blog/products/data-analytics/how-the-open-knowledge-format-can-improve-data-sharing`
- OKF repository/spec linked from the announcement.
- Karpathy LLM Wiki gist: `https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f`
- Local research note: `research/okf.md` (ignored by Git; use as context only unless moved or force-added intentionally).

**Branch:** `feat/okf-compatibility`.

## Guardrails

- Do not make OKF the native ArcKit format in the first release.
- Preserve `ARC-NNN-TYPE-vN.N.md` filenames, Document Control tables, provenance blocks, manifest behavior, and graph hooks.
- Treat OKF v0.1 as an emerging open specification, not a settled industry standard.
- Avoid claiming universal tool compatibility. Say ArcKit can emit and read OKF-shaped bundles; actual downstream behavior depends on each OKF consumer.
- Keep Google Cloud integration wording precise: the Google Cloud blog says Knowledge Catalog can ingest OKF and serve it to Google agents; ArcKit integration still needs implementation.
- Generated non-Claude extensions must come from `scripts/converter.py`, not hand edits.

## OKF Metadata Mapping

ArcKit should emit the required OKF surface plus ArcKit-specific metadata under an `arckit:` extension key.

```yaml
---
type: Architecture Decision Record
title: Use PostgreSQL for Data Storage
description: Decision record generated by ArcKit.
resource: projects/001-project/decisions/ARC-001-ADR-001-v1.0.md
tags:
  - arckit
  - architecture
  - adr
timestamp: 2026-06-18T14:30:00Z
arckit:
  document_id: ARC-001-ADR-001-v1.0
  doc_type: ADR
  project: 001-project
  category: Architecture
  version: "1.0"
  source_path: projects/001-project/decisions/ARC-001-ADR-001-v1.0.md
---
```

Field rules:
- `type`: OKF concept type. Default to ArcKit doc-type display name from `DOC_TYPES`.
- `title`: first markdown `#` heading when present; otherwise doc-type display name.
- `description`: short generated sentence from title, document id, and category.
- `resource`: repo-relative source path for local bundles; preserve external URLs when importing OKF.
- `tags`: `arckit`, lowercase doc-type code, lowercase category, and optional regime tag.
- `timestamp`: file modification time for export; current timestamp for new writes; preserve imported timestamp when round-tripping.
- `arckit.document_id`: exact ARC id without `.md`.
- `arckit.doc_type`: extracted ArcKit type code.
- `arckit.project`: project directory id, for example `001-project`.
- `arckit.category`: category from `DOC_TYPES`.
- `arckit.version`: parsed document version string.
- `arckit.source_path`: original ArcKit path.

- Create: `tests/plugin/test_export_okf.mjs`

- [ ] **Step 1: Implement deterministic exporter**

The script should:
- Accept `--project <project-id>` or `--all`.
- Default output to `okf/` under repo root.
- Walk `projects/{project}/**/ARC-*.md`.
- Emit one OKF concept markdown file per ArcKit artifact.
- Preserve markdown links where possible.
- Write `index.md` with bundle contents.
- Write `log.md` with export timestamp, source repo, command args, artifact count, and ArcKit version when available.
- Write no network calls.
- Exit non-zero on invalid project id, missing `projects/`, or write failure.

Suggested output layout:

```text
okf/
  index.md
  log.md
  projects/
    001-project/
      decisions/
        ARC-001-ADR-001-v1.0.md
      research/
        ARC-001-RSCH-001-v1.0.md
```

- [ ] **Step 2: Add `/arckit:export-okf` command**

- Explain it exports a portable OKF bundle.
- Run the script with arguments from the user prompt.
- Summarize artifact count, output path, and warnings.
