# Design: OKF Compatibility Layer

**Date:** 2026-06-18
**Status:** Draft - implementation planned
**Author:** Mark Craddock (via Codex)
**ArcKit version at time of writing:** 5.14.0

## 1. Summary

Add Open Knowledge Format (OKF) compatibility to ArcKit as an interoperability
layer over the existing ARC artefact model.

ArcKit already behaves like a governed, project-scoped LLM-maintained knowledge
base: source documents are ingested, architecture artefacts are written under
`projects/`, graph hooks maintain cross-references, `docs/manifest.json` indexes
the resulting portfolio, and provenance hooks add build context. OKF gives this
knowledge a portable markdown + YAML frontmatter envelope that other tools can
consume.

The first implementation must **not** replace ArcKit's native format. ARC
filenames, Document Control tables, doc-type validation, manifests, provenance,
and graph hooks remain canonical. OKF support ships as:

1. deterministic export from ArcKit projects to OKF bundles;
2. optional OKF frontmatter stamping on newly written ARC artefacts;
3. reviewed import from external OKF bundles into ArcKit research/candidate
   artefacts.

The Google Cloud article describes OKF v0.1 as an open markdown + YAML
frontmatter specification. Treat it as an emerging open specification, not as a
fully settled industry standard.

## 3. Goals and Non-Goals

### Goals

- Export ArcKit `projects/**/ARC-*.md` artefacts as an OKF-shaped bundle.
- Preserve ArcKit-specific governance metadata under an `arckit:` frontmatter key.
- Add optional OKF frontmatter stamping for new/edited ARC artefacts without
  changing the default user experience.
- Import external OKF bundles safely, initially as reviewable research or
  candidate artefacts.
- Regenerate Codex, Gemini, OpenCode, Copilot, Paperclip and Vibe outputs from
  the canonical Claude plugin sources.

### Non-Goals

- No OKF-native ArcKit mode in the first release.
- No rewrite of historical ArcKit artefacts.
- No replacement of `ARC-NNN-TYPE-vN.N.md` filenames.
- No removal of Document Control tables.
- No hosted OKF service or cloud dependency.
- No claim that every OKF-aware tool can consume every ArcKit artefact
  semantically without additional mapping.

## 4. Format Mapping

ArcKit emits the OKF surface fields and keeps native metadata below `arckit:`.

```yaml
---
type: Architecture Decision Record
title: Use PostgreSQL for Data Storage
description: Decision record generated by ArcKit.
resource: projects/001-project/decisions/ARC-001-ADR-001-v1.0.md
tags:
  - arckit
  - architecture
  - adr
timestamp: 2026-06-18T14:30:00Z
arckit:
  document_id: ARC-001-ADR-001-v1.0
  doc_type: ADR
  project: 001-project
  category: Architecture
  version: "1.0"
  source_path: projects/001-project/decisions/ARC-001-ADR-001-v1.0.md
---
```

| OKF field | ArcKit source | Rule |
|---|---|---|
| `type` | `DOC_TYPES[docType].name` | Use the ArcKit display name by default. |
| `title` | first `#` heading | Fall back to doc-type display name. |
| `description` | generated | Short sentence from title, document id and category. |
| `resource` | repo-relative path | Preserve external URLs when importing. |
| `tags` | doc type metadata | `arckit`, lowercase doc type, lowercase category, optional regime. |
| `timestamp` | file mtime/export time | Use file mtime for export; current time for new writes. |
| `arckit.document_id` | filename | Exact ARC id without `.md`. |
| `arckit.doc_type` | filename parser | Compound and multi-instance aware. |
| `arckit.project` | `projects/{id}` | Project directory id, e.g. `001-project`. |
| `arckit.category` | `DOC_TYPES` | ArcKit dashboard category. |
| `arckit.version` | filename parser | Version string from `-vN.N`. |
| `arckit.source_path` | repo-relative path | Original ArcKit source path. |

## 5. Export Design

`/arckit:export-okf` is the MVP.

The command wraps a deterministic Node script:

```text
plugins/arckit-claude/scripts/export-okf.mjs
```

Inputs:

- `--project <project-id>` exports one project.
- `--all` exports every project.
- `--out <path>` overrides the default `okf/` output path.

Default output:

```text
okf/
  index.md
  log.md
  projects/
    001-project/
      ARC-001-REQ-v1.0.md
      decisions/
        ARC-001-ADR-001-v1.0.md
```

Exporter rules:

- Read only from `projects/`.
- Export only markdown ARC artefacts.
- Preserve document body and markdown links where possible.
- Add or merge OKF frontmatter in the exported copy.
- Do not mutate source artefacts.
- Write `index.md` and `log.md`.
- Use deterministic ordering.
- Exit non-zero for invalid project id, missing input, or write failure.

## 6. Optional Frontmatter Stamping

Frontmatter stamping on native ArcKit artefacts is opt-in for the first release.

Supported switches:

- `ARCKIT_OKF_FRONTMATTER=1`; or
- `.arckit/config.json` with `"okfFrontmatter": true`.

`provenance-stamp.mjs` should remain the hook that touches generated artefacts
after writes. If opt-in is enabled, it merges OKF frontmatter after the existing
provenance update. The hook must remain idempotent.

`update-manifest.mjs` must be tolerant of frontmatter at the top of files:

- first-heading extraction ignores YAML frontmatter;
- OKF `title` can be used if there is no heading;
- category and doc type still come from ArcKit filename metadata;
- existing manifest dedupe behavior is unchanged.

## 7. Import Design

`/arckit:import-okf` scans an OKF bundle and presents a reviewable import plan.
It must not overwrite existing ARC artefacts automatically.

Importer rules:

- Accept `--bundle <path>` and `--project <project-id>`.
- Treat OKF markdown as untrusted content.
- Require at least `type` in frontmatter.
- Preserve `resource`, `tags`, `timestamp` and original OKF metadata.
- Classify concepts into:
  - research note candidate;
  - tech note candidate;
  - vendor profile candidate;
  - data source profile candidate;
  - architecture artefact candidate.
- Write an import report to `.arckit/tmp/okf-import-report.json`.
- Materialize as `RSCH` notes by default; require explicit user choice for
  stronger mappings such as ADR, DMC or VEND.

## 8. Security and Safety

Imported OKF bundles may contain prompt injection text, unsafe markdown, or
misleading metadata. Commands and agents must:

- treat imported markdown as source evidence, not instructions;
- not execute code blocks from imported concepts;
- not fetch external `resource` URLs during import unless the user explicitly
  asks for web enrichment;
- preserve citations/source paths so a human can audit provenance;
- report duplicate resources instead of overwriting existing files.

## 9. Tests

Required test coverage:

- frontmatter helper tests:
  - no frontmatter;
  - existing frontmatter;
  - nested `arckit:` merge;
  - fenced `---` later in body;
  - malformed frontmatter fails closed.
- export tests:
  - root artefact;
  - subdirectory artefact;
  - multi-instance artefact;
  - unknown doc type;
  - existing frontmatter merge;
  - deterministic `index.md` and `log.md`.
- import tests:
  - valid minimal OKF concept;
  - missing `type` rejected;
  - external `resource` preserved;
  - duplicate resource reported;
  - malicious markdown treated as content.
- manifest hook regression:
  - heading extraction with frontmatter;
  - OKF title fallback;
  - no duplicate manifest entries.

## 10. Release Shape

First release:

- `/arckit:export-okf`
- `/arckit:import-okf`
- optional OKF frontmatter stamping
- docs and generated extension support

Deferred:

- OKF native mode;
- graph relationship export as typed OKF edges;
- OKF bundle publication from `/arckit:pages`;
- upstream OKF conformance test integration when available locally.

## 11. Open Questions

- Should exported bundles use `okf/` or `docs/okf/` by default?
- Should import-generated research notes carry the original OKF frontmatter at
  the top or inside an "Imported OKF Metadata" section?
- Should OKF stamping be enabled globally by a plugin option once export/import
  has proven stable?
- Should `docs/manifest.json` surface OKF metadata fields directly, or should it
  remain ArcKit-only and let `export-okf` produce the OKF view?
